# Sonata Agent Harness

NPX/Bunx starter kit for generating projects with strong agent direction and low-token operation.

It borrows the useful parts of harness engineering: keep repo knowledge local, keep root instructions short, make plans first-class, expose validation commands, and convert repeated human judgment into docs or checks. Generated projects also include Sonata-specific caveman skills for compressed agent output.

## CLI Usage

From GitHub:

```bash
bunx github:gisketch/sonata init
```

With a project folder name:

```bash
bunx github:gisketch/sonata init trade-mod
```

The CLI then asks for project context. Codex is selected by default. To generate compatibility files too:

```bash
bunx github:gisketch/sonata init trade-mod --agents all
```

Retrofit an existing project without overwriting existing files:

```bash
cd existing-project
bunx github:gisketch/sonata retrofit
```

Refresh a Sonata project with only missing safe files:

```bash
cd existing-project
bunx github:gisketch/sonata update
```

Local development:

```bash
node bin/sonata.js init my-project --yes
npm run check
```

## What This Gives You

- `sonata init`: interactive project generator.
- `sonata update`: non-breaking refresh that adds missing harness files and never overwrites existing project files.
- Codex-first setup: generated `AGENTS.md`, `.codex/skills/`, and `.codex/prompts/` are always present.
- Optional compatibility: Copilot instructions and Claude Code skill files when selected.
- Generated `AGENTS.md`: short table of contents for agents.
- Generated `docs/`: project memory, architecture, quality, plans, and references.
- Generated `.github/skills/init-sonata/`: project-context initialization workflow.
- Generated `.github/skills/caveman-sonata/`: terse harness-aware implementation workflow.
- Generated `.github/skills/retrofit-sonata/`: existing-project harness normalization workflow.
- Generated `.github/prompts/`: reusable init, caveman, and review prompts.
- Generated `scripts/check-sonata.sh`: project harness structure check.

## Quick Start

1. Run `bunx github:gisketch/sonata init`.
2. Select agent targets. Codex is auto-selected; Copilot and Claude Code are optional.
3. Answer project intent, kind, stack, package manager, first milestone, and default caveman mode.
4. `cd` into the generated folder.
5. Use `/init-sonata this project is ...` to fill or refine repo context.
6. Use `/retrofit-sonata` when adapting an existing project or scattered docs into the Sonata standard.
7. Run `sonata update` later to add newly available safe harness files without overwriting local edits.
8. Use `/caveman-sonata add a module that ...` for terse implementation work.
9. Run `./scripts/check-sonata.sh` before handing the repo to another agent.

## Operating Model

Humans steer. Agents execute. The repo is the system of record.

When an agent struggles, do not only retry the prompt. Add the missing map, rule, check, or template so the next run has leverage.

## Generated Workflow

- `/init-sonata`: convert rough project intent into durable docs and first execution plan.
- `/retrofit-sonata`: inventory an existing project, preserve useful docs, and normalize it into the harness standard.
- `/caveman-sonata`: do implementation with terse progress, explicit acceptance, and validation.
- `init-sonata:`, `retrofit-sonata:`, and `caveman-sonata:` prompt labels are included for Codex-oriented workflows.
- Codex skill picker entries come from `.codex/skills/<name>/SKILL.md`.
- `docs/quality.md`: always list the real checks for the generated project.
- `docs/exec-plans/active/`: use for larger features and architecture decisions.

## Harness Fit

| Harness engineering point | Sonata coverage |
|---|---|
| Empty-repo scaffold | `sonata init` creates repo structure, docs, skills, prompts, checks, and app skeleton. |
| Humans steer, agents execute | Prompts capture intent; generated skills convert it into repo-local docs and plans. |
| Repo knowledge as source of truth | Generated `docs/` holds project brief, architecture, quality, references, and execution plans. |
| Map, not manual | Generated `AGENTS.md` stays short and links to deeper docs. |
| Progressive disclosure | Agents start at `AGENTS.md`, then read only relevant docs from `docs/index.md`. |
| Agent legibility | Source, tests, config, docs, prompts, and checks are local and inspectable. |
| Architecture and taste | `docs/architecture/index.md` defines boundaries; project-specific lint/tests should enforce them once stack is known. |
| Planning as artifact | `docs/exec-plans/active/` and `completed/` are generated by default. |
| Mechanical checks | `scripts/check-sonata.sh` validates harness structure; `docs/quality.md` records real project checks. |
| App/UI/log legibility | Generated quality docs reserve space for run, smoke, UI, log, and observability checks once the stack exists. |
| Entropy cleanup | `/retrofit-sonata` and review prompts inventory drift, re-home docs, and normalize the harness. |
| Token efficiency | Caveman rules are default for chat; durable details live in docs instead of repeated prompts. |

The starter intentionally does not invent stack-specific observability, architecture linters, CI, or test commands. `/init-sonata` and `/retrofit-sonata` add those once the real stack is known.

## Example: New Project Flow

```bash
bunx github:gisketch/sonata init trade-mod
cd trade-mod
./scripts/check-sonata.sh
```

Answer the prompts:

```text
Agent targets: Codex
what are we building?: NeoForge 1.21.1 mod about villager trading
project kind: NeoForge 1.21.1 mod
stack/runtime: Java 21 + Gradle
package manager: gradle
first useful milestone: first custom villager trade path works in dev world
default caveman mode: full
```

Then drive work through the generated harness:

```text
/init-sonata this project is a NeoForge 1.21.1 trading mod. Preserve vanilla feel. First milestone: one custom trader with configurable offers.
/caveman-sonata add the first trading domain module and update docs/quality.md with verified Gradle commands.
/caveman-sonata add tests or smoke steps for trade offer generation.
```

Expected result:

- `docs/project-brief.md` captures product intent.
- `docs/architecture/index.md` captures mod domains and boundaries.
- `docs/quality.md` lists verified Gradle commands.
- `docs/exec-plans/active/` holds any broad feature plan.
- Final handoff includes checks run, not a vague status update.

## Example: Retrofit Flow

```bash
cd existing-mod
bunx github:gisketch/sonata retrofit --agents all
./scripts/check-sonata.sh
```

Then ask the agent:

```text
/retrofit-sonata inventory this repo, preserve existing markdown, move durable project knowledge into docs/, keep README user-facing, and make it ready for Codex-first development.
```

Expected result:

- Existing docs are preserved, linked, moved, or summarized.
- Root `AGENTS.md` becomes the short map.
- Scattered architecture notes move into `docs/architecture/`.
- Build/test/run commands move into `docs/quality.md` as verified or unverified.
- Broad cleanup gets an execution plan before risky changes.
- Copilot and Claude files exist only because `--agents all` was selected.

## Why This Shape

The CLI asks a few high-leverage questions once, then writes the answers into the project. Future agent runs spend fewer tokens rediscovering context and fewer turns asking what the project is supposed to be. Codex gets the primary path; Copilot and Claude Code can join without changing the harness model.
